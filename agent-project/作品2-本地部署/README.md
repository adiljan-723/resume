# 作品2：Ollama 本地部署 — Qwen2.5 模型体验与对比

## 做了什么

在本地 Windows 电脑上通过 Ollama 部署了 Qwen2.5:7B 开源大模型，实现完全离线推理。
设计了一套对比评测方案，量化分析本地模型 vs 云端 API 的差异。

## 技术栈

- **Ollama**：本地模型管理 & 推理引擎
- **Qwen2.5:7B**：阿里通义千问开源模型（中文能力 Top 级）
- **DeepSeek API**：云端对比基准

## 核心对比数据

| 维度 | 本地 Qwen2.5:7B | 云端 DeepSeek API |
|------|-----------------|-------------------|
| 部署方式 | Ollama 本地推理 | HTTP API 调用 |
| 硬件需求 | CPU 即可（16GB RAM） | 无 |
| 首次响应延迟 | 3-8 秒 | 1-2 秒 |
| 中文理解 | ⭐⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 代码生成 | ⭐⭐⭐ | ⭐⭐⭐⭐ |
| 成本/千次调用 | ¥0 | ¥2-5 |
| 数据隐私 | 完全本地 | 数据上云 |
| 适用场景 | 高频调用、敏感数据 | 复杂推理、精度优先 |

## 面试话术

> "我在本地用 Ollama 部署了 Qwen2.5 开源模型，和 DeepSeek API 做了系统的对比评测。
> 结论是：简单问答和中文理解场景，本地模型能力接近云端，但成本为零，
> 适合做高频调用的第一层路由。复杂推理任务还是需要云端大模型。
> 这种'本地+云端混合路由'的架构，是我认为 AI 产品成本优化的最佳实践。"

## 模型选型决策框架

基于本次实践总结的决策树：

```
需求分析
├── 数据敏感？（医疗/金融/企业内部）
│   ├── 是 → 本地部署（Ollama/vLLM）
│   └── 否 → 继续判断
├── 日调用量 > 10,000 次？
│   ├── 是 → 本地模型处理 70% 简单问题 + 云端处理 30% 复杂问题
│   └── 否 → 直接走云端 API
└── 延迟要求 < 500ms？
    ├── 是 → 需要 GPU 加速本地推理（vLLM + A100）
    └── 否 → CPU 本地推理即可（Ollama）
```
